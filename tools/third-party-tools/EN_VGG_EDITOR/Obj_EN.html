📖 Object Editor Manual

Complete guide to editing Evil Islands game objects (*.mob files)

Auto-Apply: When enabled (Settings → Auto Apply), changes are automatically saved when switching to another object or field. You don't need to press Apply (F5) every time.

1. Objects

Static objects on the map: buildings, rocks, trees, furniture, etc.

Common Fields

FieldTypeDescription
IDDWordUnique numeric identifier. Must be unique for all objects on the map. Used in scripts via GetObject() as a number and via GetObjectByID() as a string.
TypeDWordObject type code. Determines how the engine processes this object.
ParentIDDWordParent object ID. Used for inheriting objects from each other — binding objects together.
PlayerByteDiplomatic group number (0-31). Determines ownership.
InScriptByteA complex parameter that affects object dynamics when working with it via script procedures and in edge cases different from basic game situations. It must be set to 1 for correct operation with objects through script procedures; for others it may be 0.
DropShadowByte1 = render shadow under the object. 0 = no shadow.
NameStringInternal name on the map. Used for readability. Case is ignored by the engine.
TemplateStringModel name from 'Res/figures.res' (e.g., 'fence00', 'stwali1').
ParentStringInternal name of the parent object. Used for readability. Case is ignored by the engine.
Texture1StringPrimary texture name from 'Res/textures.res'.
Texture2StringSecondary texture name from 'Res/textures.res'.
QuestInfoStringString with quest information. If the specified condition equals 1, the unit will be highlighted with a quest marker on the map. Can be used even without quests in the database — to highlight needed non-quest units.
Position3FloatsWorld coordinates (X, Y, Z) in WCS. Format: x, y, z
RotationQuaternionOrientation as a W,X,Y,Z quaternion. Use the XYZ fields in degrees at the bottom right for convenient rotation editing of the entity.
Complection3FloatsScale factors (X, Y, Z). 1.0, 1.0, 1.0 = normal size.
CommentsStringFree-form comment. Not used by the engine, only for notes in the editor.
PartsMultiStringList of sub-object parts. One part per line. Defined in the model's *.mod header file. If empty, all parts of the model are used simultaneously.
Tip: Rotation is stored as a quaternion (W,X,Y,Z). The default "no rotation" value is 1.0, 0.0, 0.0, 0.0. Use the Rotation XYZ fields (at the bottom right) to set rotation in degrees — the quaternion will update automatically.

2. Units

Living entities: player characters, NPCs, enemies, creatures. Units have all Object fields PLUS additional fields below.

Unit-Specific Fields

FieldTypeDescription
ImportByte1 = the unit needs to use some of the parameters described below (from "Armors" to "Stats"). 0 = everything is taken from 'Res/database.res/units.udb'.
PrototypeStringPrototype name from 'Res/database.res/units.udb' (e.g., 'Human Hero', 'Rick', 'HareF0', 'PigF0').
ArmorsMultiStringEquipped armor items. One item per line. Format: item_name.material[enchantment]
WeaponsMultiStringEquipped weapons. Same format as Armors.
SpellsMultiStringLearned spells. Format: prototype{rune;rune;...}
QuickItemsMultiStringQuick-use items (wands, scrolls). Same format as Armors.
QuestItemsMultiStringQuest items in inventory. One per line.
StatsStatsCharacter statistics.

Logic Slots

Each unit has 5 logic slots (from Logic 1 to Logic 5). Use the "Logic" dropdown to switch between:
  • Unit Properties — main unit fields (above)
  • Logic 1-5 — AI behavior configurations

3. Levers

Interactive mechanisms: doors, switches, gates, trap activation. Have all Object fields PLUS:

FieldTypeDescription
ScienceDWord3Opening requirements: quest item ID, skill amount, hand ability. Format: id, skill, flags
CurrStateByteCurrent state (0 = off, 1 = on). Use GetLeverState() in scripts.
TotalStatesByteTotal number of states the lever can have.
CycledByte1 = the lever can be toggled repeatedly. 0 = single use.
DoorByte1 = this lever controls a door. 0 = regular lever. For doors, the character will try to approach from the front or back sides, described mathematically in 'Res/database.res/levers.ldb'.
CalcGraphByte1 = recalculate the AI pathfinding graph when toggled.
Warning: For doors and other objects that affect passability, CalcGraph must be set to 1.

4. Traps

Damage-dealing or effect-triggering zones. Have all Object fields PLUS:

FieldTypeDescription
TrDiplomacyDWordDiplomatic group that the trap attacks. -1 or 0xFFFFFFFF = attacks everyone.
SpellStringSpell or effect applied when triggered. Format: prototype{rune;rune;...}
ReloadingByte1 = the trap can trigger repeatedly. 0 = single use.
CastOnceByte1 = apply the spell only once, 0 = strike all targets at once.
Areas3FloatsArray2D trigger zones. Each line: x, y, area.
Targets2FloatsArrayTarget points, initially prepared for discharging magic traps. Each line: x, y.
Trap activation: Use ActivateTrap(GetObjectByID("id"), 1) in scripts to activate, or 0 to deactivate.

5. Torches

Light-emitting objects with sound. Have all Object fields PLUS:

FieldTypeDescription
LightPoint3FloatsOffset of the light emission point from the object's position. Format: x, y, z
PowerFloatLight intensity. Positive = light, negative = darkness.
SoundStringPath to the sound file (e.g., nature\fire\small.wav). Empty = no sound.

6. Lights

Pure light sources (no physical model). Simpler than Torches.

FieldTypeDescription
IDDWordUnique identifier.
ShadowByte1 = casts shadows. 0 = no shadows.
Position3FloatsLight position (absolute). Format: x, y, z
Color3FloatsRGB color (each from 0.0 to 1.0). Format: R, G, B
RadiusFloatLight radius. Positive = light, negative = darkness.
NameStringInternal name.
CommentsStringEditor notes.
Color examples: Warm firelight = 1.0, 0.6, 0.2. Cold moonlight = 0.5, 0.6, 1.0. Green poison = 0.2, 1.0, 0.2.

Visual particle effects (fire, smoke, magic, etc.).

FieldTypeDescription

9. Unit Logics

Each unit has up to 5 logic slots. Logics define AI behavior: patrol routes, guard points, aggression, and much more.

Logic Fields

FieldTypeDescription
AlarmCondByteAlarm condition code. 0 = if sees an enemy, 1 = if attacked, 2 = if suspicion was triggered.
CallForHelpFloatRadius for calling allied units for help.
CyclicByte1 = the last patrol point transitions to the first, 0 = after the last patrol point the transition goes in reverse.
AggrModeByteAggression mode (0 - absolute aggression, 1 - aggression only as revenge, 2 - absolute fear of enemies, 3 - absolute fear of enemies and the player).
ActiveByte1 = the unit is always active, 0 = the unit is active only within about 50 meters of the current player party.
ModelDWordAI behavior model code. 0 = point, 1 = guard, 2 = patrol, 3 = sentry, 4 = player, 5 = alert guard. Point: stands in place and looks in one direction. Guard: guards the specified circle, turning to a random point when standing. Patrol: patrols strictly along the given points, if points don't contain actions — turns randomly at them. Sentry: stands in place and turns to random points. Player: control is passed to player devices or the map script module. Alert guard: does the same as a guard, but at places where alarm was triggered.
RadiusFloatGuard radius.
WaitFloatWait time at each guard point (in game ticks).
Point3FloatsCentral point for guarding or sentry duty. Format: x, y, z
UsedByte1 = this logic slot is in use.
AlarmNumbByteAlarm variable number that this logic responds to.

Guard Points

Guard points define waypoints for patrol and guard behavior. Each point has:
  • Coordinates — waypoint position (X, Y, Z)
  • Actions — what the unit does at this point

Guard Point Actions

Each guard point can have multiple actions. Format in the action editor:

x, y, z; waitTime; speed; flags
x, y, z; waitTime; speed; flags
ParameterDescription
x y zLook-at point coordinates (where the unit looks)
waitTimeHow long to stay at this action (game ticks, 15 = 1 second)
speedMovement speed to this point
flagsBehavior flags (bitmask): poorly implemented in the game, 0 - normal behavior, 1 - will rest at the point.
Important: When editing guard points, you must first select a specific logic slot (Logic 1-5) from the dropdown. Guard points belong to individual logic slots, not to the unit as a whole.
Workflow:
1. Select a unit in the list
2. Select a logic slot from the dropdown (e.g., "Logic 1")
3. Edit the logic fields (Active, Radius, Wait, etc.)
4. Click "Add Point" to create guard points
5. Select a point to edit its coordinates and actions
6. Press Save (F5) or rely on Auto-Apply

10. General Tips

Auto-Apply

When enabled (Settings → Auto Apply), any changes are automatically saved when you:

  • Click on another object in the list
  • Switch to another field
  • Switch tabs or logic slots

Adding Objects

Click "Add" to create a new object from the default template. The new object is a copy of the built-in template for that type.
